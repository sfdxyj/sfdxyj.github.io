2024-01-22 ftp下载性能优化

发表于 2024-01-22 更新于 2024-01-31 Valine：

问题描述

维保合同下载实现ftp下载功能。支持自定义配置。做完发现下载速度缓慢 30s 下载完成

问题分析

以下为下载代码

问题

  1. in = ftp.retrieveFileStream(path); 查看源码可以得出返回是个缓存流。
  2. ftp.changeWorkingDirectory(“/“); 改变目录，跳转到根目录。 根目录有权限限制，查看日志发现这行代码抛错。

优化思路

  1. 增大缓存区域 ftp.setBufferSize(1024 * 1024 * 10);。
  2. 删除ftp.changeWorkingDirectory(“/“)。

优化结果

下载速度优化到 3 秒 以内

ftp与sftp的区别

ftp是一个文件传输服务,使用20，21两个端口，20是数据链路的端口，21是控制链路的端口。
sftp也是用来传输文件的，但它的传输是加密的，是ssh服务的一部分，没有单独的守护进程，是ssh服务的一部分，可以看做是ssh服务文件传输方案。和ssh一样，使用22端口。

实现sftp 文件操作的一些类

  1. GenericObjectPool GenericObjectPool 是 Apache Commons Pool 提供的对象池，使用的时候需要调用 borrowObject 获取一个对象，使用完以后需要调用 returnObject 归还对象，或者调用 invalidateObject 将这个对象标记为不可再用。

2024-01-01-导出组件优化 hutool

发表于 2024-01-01 Valine：
本文字数： 16k 阅读时长 ≈ 28 分钟

问题描述

最近做需求，有个功能需要导出。技术用的是 hutool 工具的导出。发现大家导出都是后端根据前端页面写死了。这种写法太不灵活了，太蠢了。于是想着写个封装导出功能的工具类。由前端给出需要导出的字段。后端动态实现导出功能。

问题分析

以下是导出代码 后端写死了

优化思路

  1. 先分析hutool 导出的api。重要的api多表头合并 ExcelWriter.merge(int firstRow, int lastRow, int firstColumn, int lastColumn, Object content, org.apache.poi.ss.usermodel.CellStyle cellStyle).
    前端用的是react。 表头数据是一个树。 所以，我们需要知道一个树的深度。 这个深度就是表头的宽度。也就是合并的行数。 我们也需要知道树叶子节点的宽度。这个是确定多表头的列数。 由以上思想 先建一个column 类
  1. 再建一个操作树的工具类
  1. 导出工具类
  1. 前端请求类
  1. hutool 导出APi 文档地址

优化结果

后端不需要在写死导出表格的数据, 几行代码就解决导出。

进一步优化空间

  1. idmc 转换，运用反射技术实现。 后端不再需要去库里查询字典。
  2. 算法优化。 树的广度遍历与深度遍历都是用的递归。 还有进一步优化空间。
  3. 性能优化。 可以根据导出数据的大小。使用多线程去导出。或换成 BigExcelWriter 去导出。
    以上是我觉得还可以优化的地方。因为业务繁忙，以后有时间再去优化。

bug描述

功能正常，但点击组件出现警告信息。

bug分析

异步调用时，造成了内存泄漏。卸载组件后更新状态，执行状态更新和运行异步操作的 React 组件就会导致内存泄漏问题。
分析代码发现是以下代码导致的。

这段代码导致只要窗口变化，pageInfo 状态变化导致组件随之更新，然后就触发异步调用。导致发起大量的异步请求。当点击其他组件时，该组件卸载。导致内存泄漏。

解决方案

  1. useUpdateEffect 改成 useEffect
  2. handleOnResize 里 更改状态的代码去掉
