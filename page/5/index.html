redo 日志 undo 日志 和 binlog 日志的区别

redo 日志、Undo 日志和 Binlog 日志是 MySQL 中不同类型的日志文件，它们各自用于不同的目的，具有不同的特性和作用：
Redo 日志（重做日志）：
作用：Redo 日志是用于恢复数据库数据一致性和持久性的日志。它记录了每个事务对数据库的物理更改，包括数据页的修改。
存储位置：通常存储在 InnoDB 存储引擎中，以二进制文件的形式存在于数据库的数据目录中（例如 ib_logfile0 和 ib_logfile1）。
特性：Redo 日志是物理日志，记录了对数据的实际更改。它以追加方式写入，通常是循环使用的，以节省磁盘空间。Redo 日志的主要作用是在数据库崩溃或重新启动后，通过重播 Redo 日志中的操作来恢复数据的一致性和完整性。

Undo 日志（回滚日志）：
作用：Undo 日志用于支持事务的回滚和 MVCC。它记录了事务对数据的修改之前的旧值，以便在需要回滚事务或执行读取操作时提供原始数据的一致性视图。
存储位置：通常存储在 InnoDB 存储引擎中，以表空间的方式存在。
特性：Undo 日志记录了事务的逻辑操作，以便在需要时执行反操作（回滚）。每个事务都有自己的 Undo 日志，用于维护数据的多版本并发控制（MVCC）。

Binlog 日志（二进制日志）：
作用：Binlog 日志用于记录数据库的更改操作，包括数据的插入、更新和删除，以及 DDL（数据定义语言）操作，例如表的创建和删除。它通常用于数据复制、备份和恢复。
存储位置：通常存储在 MySQL 服务器的二进制日志文件中，以文本或二进制格式记录。
特性：Binlog 日志是逻辑日志，它记录了数据库的逻辑更改操作。它通常由 MySQL 主服务器用于将更改传送给从服务器，以实现数据复制。Binlog 日志还可以用于创建点播或增量备份。

总结：Redo 日志用于恢复数据库的物理更改，Undo 日志用于支持事务的回滚和 MVCC，Binlog 日志用于记录数据库的逻辑更改操作。这些日志在数据库管理和维护中起着不同的角色，以确保数据的一致性、完整性和可用性。

TCP 三次握手

TCP（传输控制协议）的三次握手是建立 TCP 连接的过程，用于确保通信的两端都愿意接受和发送数据。下面是 TCP 三次握手的过程：

第一次握手 - 客户端向服务器发送请求：

客户端首先创建一个用于通信的 TCP 套接字，并将标志位设置为 SYN（同步请求）。
客户端选择一个初始序列号（ISN，Initial Sequence Number）并包含在 TCP 报文中。
客户端选择一个随机的初始序列号，这个序列号用于后续的数据传输。
客户端将这个 SYN 报文发送给服务器，请求建立连接。
第二次握手 - 服务器确认请求：

服务器收到客户端的 SYN 报文。
服务器也创建一个用于通信的 TCP 套接字，并将标志位设置为 SYN。
服务器选择一个自己的初始序列号（ISN）。
服务器将自己的初始序列号和确认号设置为客户端的 ISN 加 1，并包含在 TCP 报文中。
服务器将这个 SYN+ACK 报文发送给客户端，表示接受连接请求。
第三次握手 - 客户端确认连接：

客户端收到服务器的 SYN+ACK 报文。
客户端将确认号设置为服务器的初始序列号加 1，并将标志位设置为 ACK（确认）。
客户端将这个 ACK 报文发送给服务器，表示确认连接。
连接建立：

服务器收到客户端的 ACK 报文后，连接建立。
从此时开始，客户端和服务器都可以安全地在连接上进行数据传输。
至此，TCP 连接建立完成，双方可以开始进行双向的数据传输。三次握手的过程确保了连接的双方都能够互相通信，并且双方都知道连接已经建立。在数据传输完毕后，还会进行四次握手来安全地关闭连接。这些握手过程是 TCP 协议中用于建立和终止连接的关键步骤，确保了数据的可靠传输。

TCP 4 次挥手

TCP 连接的关闭需要四次挥手的过程，是为了确保数据的可靠传输和连接的完全终止。四次挥手的目的如下：

客户端通知服务器关闭：第一次挥手，客户端发送一个带有 FIN（Finish）标志的 TCP 报文给服务器，表示它已经完成了数据的发送，希望关闭连接。客户端继续允许接收来自服务器的数据。

服务器确认关闭请求：第二次挥手，服务器接收到客户端的 FIN 报文后，发送一个带有 ACK 标志的 TCP 报文给客户端，以确认收到关闭请求。此时，服务器仍然可以向客户端发送数据。

服务器通知客户端关闭：第三次挥手，服务器决定关闭连接时，它会发送一个带有 FIN 标志的 TCP 报文给客户端，表示服务器已经完成了数据的发送，希望关闭连接。

客户端确认关闭：第四次挥手，客户端接收到服务器的 FIN 报文后，发送一个带有 ACK 标志的 TCP 报文给服务器，以确认收到关闭请求。此时，连接处于完全关闭状态，双方都不再允许发送数据。

四次挥手的过程确保了双方都有足够的时间来完成未完成的数据传输，同时也允许双方确认彼此的关闭意图。这有助于防止数据的丢失和不完整传输，同时保证连接的完全终止。如果只进行三次挥手，可能导致一方提前关闭连接，另一方仍然试图发送数据，从而导致数据的丢失。四次挥手的过程提供了更完善的连接终止机制。

过滤器和拦截器的区别

过滤器 Filter 基于 Servlet 实现，过滤器的主要应用场景是对字符编码、跨域等问题进行过滤。Servlet 的工作原理是拦截配置好的客户端请求，然后对 Request 和 Response 进行处理。Filter 过滤器随着 web 应用的启动而启动，只初始化一次。Filter 的使用比较简单，继承 Filter 接口，实现对应的 init、doFilter 以及 destroy 方法即可。

拦截器是 SpringMVC 中实现的一种基于 Java 反射（动态代理）机制的方法增强工具，拦截器的实现是继承 HandlerInterceptor 接口，并实现接口的 preHandle、postHandle 和 afterCompletion 方法。

单体架构和微服务架构的区别以及优缺点

单体架构

特点

  • 单一代码库: 整个应用程序作为一个整体进行构建和部署。
  • 集中式开发: 所有功能模块紧密耦合在一起，通常共享同一个数据库。

优点

  1. 简单性: 由于所有代码都在一个地方，开发和测试相对简单。
  2. 快速部署: 可以作为一个整体进行构建和部署，减少了部署的复杂性。
  3. 性能优化: 单体应用可以在一个进程中运行，避免了网络调用的开销。

缺点

  1. 扩展性差: 随着应用程序的增长，代码库可能变得庞大且难以管理。
  2. 灵活性低: 更新或修改某个功能可能需要重新部署整个应用。
  3. 技术债务: 难以采用新的技术栈，因为所有模块紧密耦合。

微服务架构

特点

  • 模块化设计: 应用程序被拆分为多个小的、独立的服务，每个服务负责特定的业务功能。
  • 独立部署: 每个服务可以独立开发、部署和扩展，通常通过 API 进行通信。

优点

  1. 灵活性: 可以独立地更新、部署和扩展每个服务。
  2. 技术多样性: 不同服务可以使用不同的技术栈，适合其特定需求。
  3. 故障隔离: 一个服务的故障通常不会影响其他服务。

缺点

  1. 复杂性增加: 管理多个服务的部署、监控和通信增加了系统复杂性。
  2. 分布式系统挑战: 需要处理服务间通信、数据一致性和网络延迟等问题。
  3. 部署和管理成本: 需要更复杂的基础设施和 DevOps 实践来支持独立服务的部署和运行。

总结

  • 单体架构 适合小型应用或团队，快速开发和简单管理是其优势，但扩展性和灵活性较差。
  • 微服务架构 适合大型应用或需要快速迭代的团队，提供了更高的灵活性和扩展性，但需要应对更高的复杂性和管理成本。

多态

(1)多态是指不同子类型的对象对同一行为作出不同的响应。
(2)多态性分为编译时的多态性和运行时的多态性。方法重载实现的是编译时的多态性，而方法重写（实现的是运行时的多态性。
(3)对于运行时多态，特别注意，父类引用指向子类对象，在调用实例方法时，调用的是子类重写之后的，并且不能调用子类新增的方法，对于属性和static方法来说，还是执行父类原有的。

重载和重写的区别

重写（Override）是父类与子类之间多态性的一种表现。如果在子类中定义某方法与其父类有相同的名称和参数，我们说该方法被重写 (Override)。子类的对象使用这个方法时，将调用子类中的定义，对它而言，父类中的定义如同被“屏蔽”了。

重载（Overload）是一个类中多态性的一种表现。如果在一个类中定义了多个同名的方法，它们参数列表不同，则称为方法的重载(Overload)

区别：重载实现于一个类中;重写实现于子类中。

重载(Overload)：是一个类中多态性的一种表现，指同一个类中不同的函数使用相同的函数名，但是函数的参数个数或类型不同。可以有不同的返回类型;可以有不同的访问修饰符;可以抛出不同的异常。调用的时候根据函数的参数来区别不同的函数。

重写（Override): 是父类与子类之间的多态性，是子类对父类函数的重新实现。函数名和参数与父类一样，子类与父类函数体内容不一样。子类返回的类型必须与父类保持一致；子类方法访问修饰符的限制一定要大于父类方法的访问修饰（public>protected>default>private）；子类重写方法一定不能抛出新的检查异常或者比被父类方法申明更加宽泛的检查型异常

什么是SaaS系统？

软件即服务，什么是软件即服务？
很好理解，我们以前用的OA系统、财务系统、ERP系统，都是安装在我们企业的一个服务器中，数据都是存储在本地的，访问都是通过局域网进行访问（部分可能也会通过互联网）。

性能秘诀

1.最小事务

原理: 将原大事务裁减为较小单次事务，避免整个大事务引起资源独占时间过长的问题。
适用场景：在大事务中需独占使用的资源的处理逻辑，宜将热点数据独立一个新事务， 独立于主事务外，使热点数据尽可能小、尽可能快的结束，减少资源占用的时间，提高并发能力 。

2. 乐观锁

原理：乐观锁假设不会发生并发冲突，只是在提交时检查数据的原子性。相反的，悲观锁是假定会有并发操作，互斥其他对数据的操作。乐观锁对资源占用时间代价少。
适用场景：在高并发下多个事务同时修改数据库中同一条记录、争抢同一资源的场景，如：实体状态、属性的更新、访问同一资源等场景宜用乐观锁。

3. 批量处理

原理: 对于业务类似的操作，可采用批处理模式，避免循环单实体操作，减少网络的交互次数。
适用场景：需要处理的数据规模较大的场景，宜通过批量处理的方式，减少交互次数，提高性能。

4. 分页查询

原理: 当查询的数据较多时，在服务端做分页处理，单次查询返回的数据量较少，以减轻网络传输的压力。
适用场景：对查询返回结果的数据量较大的场景，宜通过服务端分页，减少返回的数据量，提高查询性能。

5. 异步处理

原理: 异步执行，提升请求的响应速度 。
适用场景：事务独立性强、耗时较长的业务过程。例如：汇总 、计提折旧、上报下发、异步集成等，宜异步处理。

6. 并行处理

原理: 采用多线程、并行计算 ，充分利用 CPU 等硬件资源，提升性能。
适用场景：同一业务场景需要处理多个业务过程，且业务处理过程无必要关联时，宜采用并行处理。

7.分区分表

原理: 由于直接读写存量数据过多的大表所导致的IO 、内存压力，会给系统带来较大的性能问题 。对于该类表，可分区或者分表存储。
适用场景：单表数据量较大，数据列较多，由于系统IO 、内存等因素引起并发写入、查询响应慢等性能问题突出的场景宜用分区分表。

8.按索引查询

原理: 索引是对数据库表中一列或多列的值进行排序的一种结构，使用索引可快速访问数据表的特定信息 。创建索引的一个主要目的就是加快检索表的数据。
适用场景：存量数据规模较大的业务数据查询。

9.数据冗余

原理: 在一个数据集合范围内重复的数据称为数据冗余。特定场景下，可通过通过冗余存储方式，降低查询复杂度，减少交互次数，提升性能。
适用场景：基础数据变化频率不高，数据同步时效要求不高，冗余数据能够提高性能。（两张表，需要查询的表冗余另一张表的一些字段，方便查询）

10.预处理数据

原理: 对于处理步骤复杂（或数据量大，重复利用度较高的中间数据，可抽象成预置数据， 使该数据消费者减少复杂度和数据吞吐量，从而提升系统的响应性能。
适用场景：业务数据关联存储的表多，处理逻辑复杂的查询、统计分析等功能，宜预处理为预置数据 。

  1. 场景描述：
    分布式光伏发电项目在并网完成后进行发电，财务部门每月按照营销提供的项目发电量与上网电量进行结算账务处理。
    分布式光伏结算-结算处理-结算确定查询功能。据实施地实际使用反馈，分布式光伏结算的结算处理查询很慢，无法满足日常的正常使用，
    同时结算处理是分布式光伏结算业务的核心功能，更需要进行优化，达到性能测试指标的要求，让使用者对分布式光伏结算应用的性能有一个较大较好的改观。
  2. 性能提升方案：预加工数据：
    未结算数据集成过来后，进行预处理，将未结算的数据写入到未结算表中，进行价格及数据异常状态的判断处理。

11.读写分离

原理: 通过对数据读写操作的分类访问，分散和降低数据库访问压力，均衡IO ；并可以针对读和写的特点优化其性能。
适用场景：对于读写不均衡，读多变化少的数据访问存在性能问题的，推荐采用读写分离模式，例如：待办事项等，宜进行读写分离。

  1. 读写分离
    将读操作和写操作分离到不同的数据库上，避免主服务器出现性能瓶颈；主服务器进行写操作时，
    不影响查询应用服务器的查询性能，降低阻塞，提高并发； 数据拥有多个容灾副本，提高数据安全性，
    同时当主服务器故障时，可立即切换到其他服务器，提高系统可用性；
  2. 读写分离原理(Mysql)
    第一步：Master（主服务器）将操作记录到binary log（二进制日志文件当中）。即每个事务更新数据完成之前先把操作记录在日志文件中，Mysql将事务串行的写入二进制日志文件中，写入日志文件完成之后，Master通知存储引擎提交事务(注：对数据的操作成为一次二进制的日志事件[binary log event]);
    第二步：slave（从服务器）把binary log拷贝到relay log（中介日志）相当于缓存作用，存储在从服务器的缓存中，首先slave会开始一个工作线程（I/O线程），I/O线程会在Master上打开一个普通的连接，然后读取binary log事件，如果已经跟上master，就会睡眠，并等待Master产生新的事件，I/O线程将读取的这些事件写入到relay log；
    第三步：slave从做中介日志事件（relay log），sql线程读取relay log事件并执行更新从服务器上的数据，使其与Master上的数据一致。
    　　总结：主服务器把操作记录到binary log——>从服务器将binary log中的数据同步到relay log（中介日志中）——>从服务器读取中介日志执行同步数据

12.应用缓存

原理: 为减少服务端访问数据库的交互，服务端缓存变化频率低的热点数据， 减轻数据库访问压力。
适用场景：对于变动率低、频繁读取的体系类数据，通过缓存方式，避免频繁访问数据库。例如：财务组织、行政组织、科目、分类、岗位、权限体系等，宜用缓存。

13.按需取数

原理: 根据业务的需要，分析其需的最小数据范围，排除与本次业务操作无关数据，以减少不必要的数据处理与数据传输。
适用场景：数据实体的字段较多或者数据量大的场景。宜按需要取数据。

14.模型封装

原理: 对于一些复合模型，如果在客户端组装相关属性需要和服务端交互多次。可在服务端处理合并客户端模型减少网络交互次数。
适用场景：客户端ID名称转换、关联数据的集中展示等，影响到数据加载、转换的性能及用户体验的，宜进行模型封装。

15.分层加载

原理: 客户端需要获取有较大量的数据，但其展示或者计算时只使用到部分数据，可逐层按需加载数据。该方式也是一种延迟加载（ lazy load ）。
适用场景：有层级关系、主从关系、先后顺序的实体查询， 宜分层加载。

redis 数据类型

  3. set String类型的无序集合
