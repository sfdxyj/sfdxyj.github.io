2024-01-01-导出组件优化 hutool

发表于 2024-01-01 Valine：
本文字数： 16k 阅读时长 ≈ 28 分钟

问题描述

最近做需求，有个功能需要导出。技术用的是 hutool 工具的导出。发现大家导出都是后端根据前端页面写死了。这种写法太不灵活了，太蠢了。于是想着写个封装导出功能的工具类。由前端给出需要导出的字段。后端动态实现导出功能。

问题分析

以下是导出代码 后端写死了

优化思路

  1. 先分析hutool 导出的api。重要的api多表头合并 ExcelWriter.merge(int firstRow, int lastRow, int firstColumn, int lastColumn, Object content, org.apache.poi.ss.usermodel.CellStyle cellStyle).
    前端用的是react。 表头数据是一个树。 所以，我们需要知道一个树的深度。 这个深度就是表头的宽度。也就是合并的行数。 我们也需要知道树叶子节点的宽度。这个是确定多表头的列数。 由以上思想 先建一个column 类
  1. 再建一个操作树的工具类
  1. 导出工具类
  1. 前端请求类
  1. hutool 导出APi 文档地址

优化结果

后端不需要在写死导出表格的数据, 几行代码就解决导出。

进一步优化空间

  1. idmc 转换，运用反射技术实现。 后端不再需要去库里查询字典。
  2. 算法优化。 树的广度遍历与深度遍历都是用的递归。 还有进一步优化空间。
  3. 性能优化。 可以根据导出数据的大小。使用多线程去导出。或换成 BigExcelWriter 去导出。
    以上是我觉得还可以优化的地方。因为业务繁忙，以后有时间再去优化。

参考资料

bug描述

后端返回前端，分页页码总数信息丢失，导致前端不能分页。

bug分析

使用的pagehelper分页插件进行分页。在进行性能优化时，把排序放到代码里排序，导致分页信息丢失。

使用PageHelper 插件后 eneRmAlarmDetails 返回的其实是page 对象，里面存有分页的信息，如果直接对其排序，会导致分页信息丢失。

解决方案

知道其原理后，只要在new 一个对象存储分页对象，排序后在把数据塞进去。

2023-11-07 sql优化处理

问题描述

能耗告警模块查询详请很慢

问题分析

定位到以下sql

发现该sql进行多表关联，且查询的字段用了函数计算。 ene_rm_alarm_details 是大表300多万数据。进行explain分析。
进行了全表扫描。 另外order by 字段没建立索引，导致索引失效

优化思路

1.拆分sql 语句, 先查出ene_rm_alarm_details的主键 id，然后在通过id去查详情数据。
2.ene_rm_alarm_details 的room_id字段上建立索引。
3.去掉order by a.alarm_time,a.room_name，改为程序进行排序。

优化结果

查询耗时1分多钟优化到 5~8 秒

进一步优化空间

业务上进行优化 ene_rm_alarm_details是大表，不应该检索所有数据。业务上应该默认查询最近一年的数据或者给一个时间范围。通过在告警时间上建立索引。把大表变成小表，然后在跟其他表进行关联时。性能会提升一大截。
